Atlas Semantic and Interaction Architecture

According to the current project docs from May 26, 2026, the Atlas problem is not lack of content. The archive already has the worlds, projects, emotional territory, visual language, and philosophical continuity. The real task is making the intelligence legible without flattening it into some miserable SaaS knowledge graph wearing a museum costume. The docs define the archive’s current phase as hierarchy, pacing, evidence, sequencing, clarity, orientation, and editorial refinement, with Atlas legibility and relation taxonomy as active priorities.

The Atlas should become the archive’s relational intelligence layer. Not a map of everything. Not a galaxy wallpaper. Not “look, lines.” The Atlas should answer one question with precision:

Why do these works belong together?

The system below turns Atlas into a navigable cultural intelligence system built around relation types, node hierarchy, weighted ties, modes of reading, and narrative traversal.

--------------------------------------------------------------------------------

PAPAZIAN ARCHIVE ATLAS

Semantic + Interaction Architecture

#### The core operating model should remain:

Enter → Trace → Read

Enter a work. Trace its relationships. Read its deeper argument. The current docs already identify this as the real interaction model beneath the visible modes, which means Atlas should not try to replace Works, Rails, or Essays. It should do one thing: let users trace meaning.

The Atlas is therefore not the “everything view.”

Works = curated entry.

Index = retrieval.

Rails = evidence sequence.

Essays = intellectual depth.

Atlas = relation logic.

That distinction matters because right now the problem is exactly that Atlas feels intelligent but does not explain itself. The audit says Atlas has lines but lacks labels, anchors, clusters, legends, relationship types, cluster names, domain boundaries, selected anchors, visual hierarchy, minimap/orientation, and relationship explanation.

#### So the new Atlas needs three simultaneous layers:

A semantic data model

that describes relations.

A visual grammar

that makes those relations readable.

An interaction model

that teaches the user what the relation means.

1. RELATION TYPES

The current prompt’s starting categories are useful, but too broad. “Memory,” “political,” “archive,” and “methodology” are not yet relation types. They are clouds of meaning. Delightful, yes. Also vague enough to make a curator quietly leave the room.

A relation type should describe

why one node is connected to another

#### The Atlas relation taxonomy should have three levels:

Primary relation class

: the broad family.

Relation type

: the specific logic of connection.

Relation claim

: the human-readable explanation shown in the UI.

A. Lineage Relations

These define historical, conceptual, or methodological descent.

Use when one project directly generates, transforms, or anticipates another.

One project begins a larger logic.

Example: Mashrou’ Leila → Systems Choreography.

evolves\_into

A project mutates into another form.

Example: Space Time Tuning Machine → DERIVE.

An earlier work anticipates a later system without directly becoming it.

Example: Architecture in Low Res → The Autopsy / Beirut Phantom.

inherits\_from

A later project explicitly carries a previous project’s method or problem.

counterpart\_to

Two works mirror each other across medium or scale.

breaks\_from

A later work rejects or corrects the limits of an earlier one.

Plain-language UI: “This came before that.”

B. Method Relations

These describe shared operating procedures.

shared\_method

Projects use the same structural logic.

tuning\_rule

Projects use deliberate constraint, delay, rupture, silence, or non-resolution.

force\_proximity

Projects organize knowledge through adjacency rather than hierarchy.

somatic\_input

The body becomes the signal source.

adversarial\_protocol

The work uses tests, audits, or verification against machine flattening.

subtractive\_logic

The work removes, withholds, degrades, or interrupts rather than adds.

procedural\_fiction

Narrative behaves as a system, interface, law, bureaucracy, or protocol.

These map directly to the project’s broader Systems Choreography premise, where architecture, music, code, narrative, interface, performance, and image are treated as interoperable systems rather than isolated objects.

2. NODE HIERARCHY

The current documents make the hierarchy issue painfully clear: not all projects should be equally expanded. The archive needs hierarchy, with Tier A flagship rails, Tier B secondary systems, and Tier C archive fragments.

Atlas hierarchy should not be identical to Works hierarchy. Works stays curated. Atlas can carry more granularity. The taxonomy docs explicitly say Atlas can carry more granularity than Works, while Works should remain editorially selective.

So Atlas should use its own node roles.

A. Lead Nodes

These are gravitational anchors. They remain visible at nearly all zoom levels.

Role: define major practice worlds and attract related nodes.

Always labeled at rest.

Largest scale.

Strongest halo.

Can open full project rail.

Show 5-9 primary relations on selection.

Display “why this matters” in the relation drawer.

B. Secondary Nodes

These are substantial projects that connect anchors.

Role: bridges, methods, research systems, specialized projects.

Medium scale.

Label on hover or when selected cluster is active.

Can open a starter rail or compressed dossier.

Show 3-5 primary relations.

May become lead nodes later.

DERIVE, HAH-WAS, Resonance Atlas, STORYLINES, Codeverse Explorer, Beirut Phantom, Photon+, Kardia, 3D Beat Synth, Music Engines.

C. Archive Fragments

These are evidence, subprojects, artworks, texts, diagrams, records, performances, or research fragments.

Role: deepen meaning without overloading Works.

Small scale.

Hidden at initial zoom except around selected parent.

Label only on hover/focus.

Open a compact object card, not a full rail.

Connect primarily to parent plus 1-2 cross-relations.

Ibn El Leil, Roman, Raasuk, SparrowOS, Tuning Rules, Mobius Engine, Drone-Bird, Localization Gap Audit, Ghost City Data Sculpture, Body Maintenance Log, 1000 Strings at Rest.

These exist in the data model but are not surfaced by default.

Role: preserve backend structure without confusing public users.

Private research.

Unverified candidates.

Weakly documented works.

Deprecated experiments.

Internal frameworks not ready for public interpretation.

Invisible by default.

Visible only in Curator Mode or Researcher Mode.

Can appear as ghost nodes when they explain a selected relation.

No public deep link unless status changes.

These are child nodes under parent systems.

Role: give Atlas more granularity than Works without expanding Works into chaos.

Appear as orbiting micro-nodes around parent when parent is selected.

Role: prevent first-view overload while rewarding traversal.

3. RELATION WEIGHTING

Every relation should have a weight and a confidence score. Weight says how important the relation is. Confidence says how well-supported it is.

Relation Strength Classes

Strong Ties

Definition: direct, explicit, evidence-backed relations.

Mentioned in project copy or docs.

Visible in rail evidence.

Direct parent/child relation.

Shared dataset, system, institution, or historical lineage.

Editorially confirmed.

Brighter line.

More persistent.

Label appears quickly.

Included in default selected-neighborhood view.

Mashrou’ Leila → The Cost of Being Queer and Arab.

STTM → DERIVE.

Localization Gap → HAH-WAS.

Medium Ties

Definition: meaningful but not primary.

Shared method, domain, or theme.

Not direct lineage.

Helps user understand practice structure.

Visible on hover/selection.

Medium opacity.

Relation label available in drawer.

Kardia → 3D Beat Synth through body-as-signal logic.

Definition: low-intensity relation useful for discovery but not main interpretation.

Shared domain only.

Shared material but no strong conceptual connection.

Same phase or medium.

Hidden by default.

Visible only when filters are active.

Thin dotted line or faint arc.

Two projects both using WebGL without deeper conceptual relation.

Latent Ties

Definition: inferred or emergent relation not yet editorially confirmed.

Semantic similarity.

Embedding proximity.

Shared metadata.

AI-assisted suggestion.

Not yet approved.

Dashed or ghosted.

“Inferred” label.

Never shown as fact.

Researcher Mode first; Curator Mode optional.

An inferred relation between SUNBURN and TEBR via environmental sonification and future folklore, pending editorial confirmation.

Temporary Ties

Definition: relation that depends on user-selected path, route, or filter.

Appears during a guided journey.

Appears while comparing two clusters.

Appears because a user has selected a theme.

Animated trace line.

Fades when path ends.

Marked as “route relation,” not permanent archive relation.

Historical Ties

Definition: chronological or career-evolution relation.

Date-based.

Project evolution.

institutional timeline.

practice phase shift.

Directional arrow.

Timeline overlay available.

Can be filtered by decade/phase/year.

Inferred Ties

Definition: machine-suggested relation.

Important distinction: latent ties may be editorially sensed but not proven; inferred ties are generated computationally.

Embedding similarity.

Shared tags.

Repeated language.

Similar artifacts.

Common references.

Never shown in default public mode.

Researcher Mode only.

Requires approval before promotion.

Weight Formula

#### Each relation should calculate a display weight:

relationWeight =
  (editorialImportance \* 0.35) +
  (evidenceStrength \* 0.25) +
  (semanticSpecificity \* 0.15) +
  (nodeTierGravity \* 0.10) +
  (pathRelevance \* 0.10) +
  (userFocusBoost \* 0.05)

editorialImportance

Curator-defined. This protects the archive from becoming an algorithmic nonsense machine. A wise precaution. Machines are very good at finding patterns and very bad at knowing when they are stupid.

evidenceStrength

Does the relation have images, rail cards, citations, artifacts, public proof?

semanticSpecificity

“Both are sound” = low.

“Both treat maqam flattening as AI evidence” = high.

nodeTierGravity

Lead-to-lead or lead-to-secondary gets more importance than fragment-to-fragment.

pathRelevance

Boosted when user is in a narrative route.

userFocusBoost

Boosted during hover, selection, or filter.

Relation Confidence

#### Separate from weight:

confidence: 'confirmed' | 'curated' | 'inferred' | 'speculative'

#### Definitions:

: backed by source material or project metadata.

: intentional editorial relation, plausible and approved.

: machine-suggested or metadata-derived.

speculative

: future-facing or conceptual, useful but not factual.

#### Display rule:

Public Atlas defaults to confirmed + curated only.

Researcher Mode can show inferred.

Curator Mode can show speculative.

Atlas = relationship tracing only.

The docs warn against turning the chromatic system into a rainbow taxonomy and recommend using colors as small accents, relation-line accents, badges, hover glows, minimap categories, and chapter markers while preserving the dominant black-field world.

Large, persistent, labeled by default.

Small, no default label.

E. Relation Governance

Every relation should have an editorial status.

relation.reviewStatus:
  | 'draft'
  | 'approved'
  | 'needs-evidence'
  | 'deprecated'
  | 'private';

Public mode only shows approved.

Curator Mode can show needs-evidence.

Researcher Mode can show draft/inferred.

Deprecated relations are hidden but retained for history.

Every strong relation needs a relation claim.

No relation without a sentence. This is the law. A line without a sentence is decoration pretending to be thought.

F. Atlas Import Workflow

#### Recommended pipeline:

Import node CSV.

Normalize domains.

Assign world cluster.

Assign tier.

Assign parent/child relation.

Generate candidate relations.

Curator approves relation claims.

Assign weight/confidence.

Render public Atlas subset.

Keep inferred graph behind Researcher Mode.

This matches the current guidance to keep the curated Works spine stable, add domain/gravity/myth-copy fields only after schema review, normalize temporal states, treat duplicates as multi-domain facets, collapse subnodes under parent projects, and allow Atlas more granularity than Works.
